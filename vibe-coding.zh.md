## 什么是 Vibe Coding？

**Vibe Coding** 是一种协作开发方式——你将 AI 助手作为技术伙伴，通过持续对话，以迭代方式构建软件。

### AI 是你的技术伙伴

**不是魔法棒** — 而是一个对话伙伴，它会：
- 建议实现思路
- 解释权衡取舍
- 执行你的决策
- 审查你的计划
- 帮助你理清问题

### Vibe Coding 循环

```
1. 明确目标
   ↓
2. 探索方案（重要！）
   ↓
3. 提供上下文
   ↓
4. AI 建议/实现
   ↓
5. 你审查并测试
   ↓
6. 对话：精炼、质疑、迭代
   ↓
7. 下一个小步骤
```

### 找到"节奏"

在以下之间保持平衡：
- 足够的上下文 ↔ 不要过于冗长
- 具体明确 ↔ 留有创意空间
- 速度 ↔ 质量
- 接受建议 ↔ 坚持你的愿景

## 思维模式的转变

Vibe Coding 不是为了更快地生成代码，而是改变你把认知资源花在哪里。

传统开发：你同时拥有架构和实现细节的所有权。

规模化 Vibe Coding：你拥有架构、接口和约束。AI 在这些边界内处理实现细节。你的工作变成了**指挥、审查和集成**，而不是编写。

这个转变听起来简单，实则不然。大多数开发者在 Vibe Coding 上失败，不是因为不会提问，而是没有内化哪些地方需要人类判断：

- **你定义系统该做什么** — AI 无法推断你的业务需求
- **你拥有架构** — 如果你不提供架构，AI 会自己发明一个，等到 10 万行代码时你就后悔了
- **你验证正确性** — AI 生成的是看起来合理的代码，不一定是正确的代码
- **你承载上下文** — 模型在会话之间会忘记一切；你不会

最能把 AI 用到极致的开发者，是那些越来越擅长**写规格说明**的人，而不是最信任 AI 输出的人。

## 哪里有效，哪里会崩

### 高杠杆的使用场景

- **样板代码和脚手架** — CRUD 处理器、序列化器、CLI 参数、配置解析器
- **测试生成** — 对定义清晰的函数生成单元测试、表驱动测试用例、Mock 脚手架
- **独立、类型清晰的函数** — 任何有明确输入、输出且无隐式依赖的代码
- **目标明确的重构** — "将这段逻辑提取为符合此签名的函数"
- **格式转换** — 数据格式互转、从描述生成 SQL、从类型生成 OpenAPI 规范
- **文档** — 从代码生成 docstring、README 章节、变更日志

### 容易出问题的场景

- **横切关注点** — 认证、日志、错误处理策略，这些必须在整个代码库中保持一致
- **复杂状态管理** — 任何正确性依赖于变更历史的代码
- **新颖算法** — AI 会生成能编译且看起来合理的内容，但可能存在细微错误
- **安全关键路径** — 输入验证、认证、加密、权限边界
- **需要理解业务领域的代码** — 业务规则、数据不变量、训练数据中不存在的监管约束
- **大规模重构** — AI 可以协助执行步骤，但无法为生产系统设计安全的迁移路径

**经验法则：** 如果你自己都写不出规格说明，AI 也很难正确实现它。

## 核心原则

### 1. 组件分离
**单一职责，清晰接口**
- 让 AI 保持专注
- 更易测试
- 更易理解

### 2. 增量开发
**微小变更，代码始终可运行**
- 5 分钟规则：如果验证需要超过 5 分钟，改动太大了
- 立即测试
- 频繁提交

### 3. 探索方案
**决策之前先获取多种思路**
- 询问替代方案
- 比较优缺点
- 做出有据可查的决策

### 4. 会话管理
**冷启动提供上下文，热启动保持流畅**
- 新会话：提供完整上下文
- 活跃会话：保持连续性
- 为下次保存会话摘要

### 5. 备份与计划
**安全第一，先思考再写代码**
- 改动前始终备份
- 创建实现计划
- 与 AI 一起审查
- 按步骤实现

## 先做架构，再 Vibe

在 Vibe Coding 大型系统时，最大的错误是从实现开始。

AI 非常擅长填充空间。如果你不给它架构，它会给你一个——来自训练数据中最常见的开源项目。那可能不符合你的需求、你的现有模式或你的运营约束。

**在任何 AI 辅助实现之前，先做这些：**

1. **定义模块边界** — 每个组件的职责范围；写下来
2. **先写接口定义** — 类型、函数签名、API 契约；这些成为 AI 必须满足的约束
3. **记录你的不变量** — 什么必须始终为真？哪些失败模式是绝对不能接受的？
4. **建立错误处理规范** — 错误如何传播、什么要记录日志、什么要暴露给调用方；不指定的话，AI 会随意发明
5. **有意识地命名** — AI 把名字当作语义提示；模糊的名字产生模糊的实现

一个有用的练习：在生成任何代码之前，先为你要构建的模块写 docstring 或规格注释。如果你写不清楚，说明你还没准备好实现它——无论有没有 AI。

## 探索多种实现方案

**黄金法则：** 当有多种实现方式时，在决策之前始终先探索选项。

### 为什么重要

- ❌ 第一个建议对你的场景未必最优
- ❌ 你可能错过更简单的替代方案
- ❌ 权衡取舍并不总是显而易见
- ❌ 正确选择取决于具体场景

✅ **解决方案：** 系统地探索选项

### 选项探索流程

**第一步：询问选项**

```
你："我需要【解决问题】。有哪些不同的实现思路？"

AI："以下是主要方案：

     方案 1：【思路 A】
     - 描述：【是什么】
     - 适合：【场景】
     
     方案 2：【思路 B】
     - 描述：【是什么】
     - 适合：【场景】
     
     方案 3：【思路 C】
     - 描述：【是什么】
     - 适合：【场景】
     
     你倾向于哪个方向？"
```

**第二步：请求优缺点对比**

```
你："针对我的具体场景对比这些方案：
     - 【约束 1】
     - 【约束 2】
     - 【优先级】
     
     给出各方案的优缺点。"

AI："针对你的场景：

     方案 1：【思路 A】
     优点：
     ✅ 【优势 1】
     ✅ 【优势 2】
     缺点：
     ❌ 【劣势 1】
     ❌ 【劣势 2】
     ..."
```

**第三步：不确定时深入探讨**

```
你："我在方案 A 和 B 之间犹豫，
     能详细说说【具体方面】吗？"

AI：[对该方面进行详细比较]

你："如果【我的情况发生变化】，哪个更好？"

AI：[解释各方案在该场景下的表现]

你："给我看看每种方案的简单示例。"

AI：[提供代码示例供对比]
```

**第四步：做出有据可查的决策**

```
你："基于【理由】，我们选方案 B。
     从最简单的实现开始。"

AI：[实现所选方案]
```

### 真实例子：缓存策略

**不好的方式：**
```
你："给这个函数加缓存。"
AI：[实现一种缓存方案]
你：[可能并不是最合适的]
```

**好的方式：**
```
你："我需要加缓存来减少重复计算，有哪些选择？"

AI："主要缓存策略：

     1. 内存缓存（dict）
        - 最快
        - 重启后丢失
        - 受内存限制
     
     2. LRU 缓存（functools）
        - Python 内置
        - 自动淘汰
        - 大小有限制
     
     3. Redis 缓存
        - 持久化
        - 跨进程共享
        - 需要 Redis 服务器
     
     4. 文件缓存
        - 持久化
        - 简单
        - 比内存慢
     
     你的使用场景是什么？"

你："单进程，缓存耗时计算，重启丢失可以接受，
     希望自动管理大小。"

AI："那 LRU 缓存（方案 2）非常合适：
     
     优点：
     ✅ 内置，无依赖
     ✅ 自动管理大小
     ✅ 简洁的装饰器语法
     ✅ 线程安全
     
     缺点：
     ❌ 重启丢失（你说可以接受）
     ❌ 不跨进程共享（单进程，没问题）
     
     要看实现吗？"

你："要，给我看看。"

AI：[实现 LRU 缓存]
```

### 何时探索选项

**以下情况务必探索：**
- 🔍 存在多种有效方案
- 🔍 性能很重要
- 🔍 可扩展性是考虑因素
- 🔍 你对该领域不熟悉
- 🔍 决策有长期影响

**可以跳过探索的情况：**
- ✓ 实现非常简单
- ✓ 标准模式直接适用
- ✓ 你已经知道最佳方案
- ✓ 后续容易修改

### 选项探索模板

**记住这个模式：**

```
你："我需要【目标】，有哪些选择？"

AI：[列出 3-5 个选项及简要描述]

你："针对我的情况对比这些选项：【约束】，
     给出优缺点。"

AI：[详细对比]

你：[如仍不确定]"详细说说【具体方面】"
     或"给我看方案 X 和 Y 的示例"

AI：[提供说明]

你："因为【理由】，我们选【方案】。"

AI：[实现]
```

## 组件分离与清晰接口

**黄金法则：** 保持组件分离、接口清晰，避免让 AI 产生混乱。

### 为什么重要

紧耦合让你和 AI 都感到困惑：
- ❌ 职责不清
- ❌ 关注点混杂
- ❌ 难以测试
- ❌ AI 建议准确性降低

✅ **解决方案：** 一个组件，一个职责，清晰的接口

### 分离策略

**1. 单一职责**

❌ **不好的方式：**
```python
class DataProcessor:
    def process(self, data):
        # 验证、转换、保存、通知、日志记录...
        # 职责太多！
```

✅ **好的方式：**
```python
class DataValidator:
    def validate(self, data) -> ValidationResult:
        """只负责验证"""

class DataTransformer:
    def transform(self, data) -> TransformedData:
        """只负责转换"""

class DataRepository:
    def save(self, data) -> SaveResult:
        """只负责保存"""
```

**2. 清晰的接口边界**

❌ **不好的方式：**
```python
def get_data(self):
    return self.db_connection  # 泄露实现细节
```

✅ **好的方式：**
```python
def get_data(self) -> List[DataRecord]:
    """返回干净的数据，隐藏获取方式"""
    return [DataRecord(...) for row in results]
```

**3. 最小化依赖**

❌ **不好的方式：**
```python
class ComponentA:
    def __init__(self):
        self.b = ComponentB()  # 紧耦合
        self.c = ComponentC()
```

✅ **好的方式：**
```python
class ComponentA:
    def __init__(self, data_provider: DataProvider):
        self.provider = data_provider  # 接口，非具体实现
```

### 焦点窗口技术

**每次只专注于一个组件：**

```
会话 1：设计并实现 DataSource
会话 2：测试 DataSource
会话 3：设计并实现 Validator
会话 4：测试 Validator
...以此类推
```

**每次会话中：**
```
你："今天我们专注于【组件 X】。
     这是它的接口：[粘贴]
     不用管其他组件。"

AI：[专注于组件 X]
```

### 组件设计检查清单

✅ **清晰的输入/输出**
```python
def process(self, input: InputType) -> OutputType:
    """明确什么进、什么出"""
```

✅ **无副作用**（尽可能）
```python
def transform(self, data: Data) -> TransformedData:
    """返回新数据，不修改输入"""
```

✅ **自包含**
```python
class Component:
    """内部包含所需的一切"""
    def __init__(self, dependencies: Dependencies):
        self.deps = dependencies
```

✅ **有文档说明目的**
```python
class DataValidator:
    """
    验证数据记录。
    
    职责：检查字段、类型、约束
    非职责：转换、保存或通知
    """
```

### 警示信号

🚩 "这个类做了太多事" → 拆分它
🚩 "测试前要设置一堆东西" → 添加接口
🚩 "修改 X 会破坏 Y 和 Z" → 解耦
🚩 "AI 总是混淆职责" → 明确边界

## 增量开发

**黄金法则：** 做微小的增量改动，避免大型重构。

### 为什么重要

大改动会带来问题：
- ❌ AI 失去上下文
- ❌ 难以验证
- ❌ 不好调试
- ❌ 可能引入多个 Bug
- ❌ 难以回滚

✅ **解决方案：** 改动小到显而易见的正确

### 5 分钟规则

**如果验证一个改动需要超过 5 分钟，它就太大了。**

❌ **太大：**
- "重构整个认证系统"
- "重写数据库层"

✅ **合适大小：**
- "添加一个密码哈希的辅助函数"
- "添加一个新的数据库查询方法"

### 增量策略

**1. 每次只做一件事**

❌ **不好的方式：**
```
你："添加缓存、改进错误处理、优化算法"
AI：[做大量改动，破坏代码]
```

✅ **好的方式：**
```
会话 1："为最后一个结果添加简单缓存"
[测试——通过！]

会话 2："让缓存大小可配置"
[测试——通过！]

会话 3："添加缓存过期机制"
[测试——通过！]
```

**2. 始终保持代码可运行**

❌ **不好的方式：**
```
你："重写这个组件"
AI：[重写所有内容]
[什么都不工作，调试数小时]
```

✅ **好的方式：**
```
你："用新模式添加新方法，保留旧方法"
[测试——两者都正常工作！]

你："将一个调用者迁移到新方法"
[测试——仍然正常！]

你："再迁移一个调用者"
[测试——仍然正常！]

...直到完成，再删除旧方法
```

**3. 婴儿步骤**

不要这样做："重写验证系统"

拆分为：
1. 添加邮箱验证规则
2. 测试邮箱验证
3. 添加电话验证规则
4. 测试电话验证
5. 提取公共逻辑
6. 测试现有验证仍然正常

### 对话模式

```
你："这个改动看起来很大，能拆分成更小的步骤吗？"

AI："可以：
     1. 【小步骤 1】- 5 分钟
     2. 【小步骤 2】- 5 分钟
     3. 【小步骤 3】- 5 分钟"

你："好，我们只做第 1 步。"

AI：[实现第 1 步]

你：[测试——通过！]

你："很好，现在第 2 步。"
```

## 会话管理

**黄金法则：** 新会话提供完整上下文，活跃会话保持流畅连续。

### 理解会话类型

**热启动（活跃会话）：**
- AI 有完整的近期上下文
- 快速，保持流畅
- 适合：迭代开发、调试
- 限制：约 10-20 轮对话

- AI 没有记忆
- 需要重新建立上下文

**黄金法则：** 先备份，制定计划，与 AI 一起审查，再按步骤实现。

- ❌ 可能误解需求
- ❌ 可能引入 Bug
- ❌ 改动可能破坏代码
- ✅ 需要对比新旧版本
- ✅ 需要在必要时回滚

**第一步：备份**

```
你："请审查这个计划：

## 为 AI 辅助构建项目结构

大型代码库有一个上下文问题：任何给定任务的相关上下文散布在许多文件中，而 AI 的上下文窗口是有限的。

设计你的代码库，让只能看到几千个 token 的模型也能导航。

### 文件和模块设计

- **保持文件小而专注** — 1500 行的文件对人类来说难以推理，对模型来说更是无法完整放入上下文
- **一个文件一个概念** — 不要在同一个模块中混合不相关的抽象
- **显式导入优于隐式全局变量** — AI 能看到导入的内容；它无法推断隐藏的全局变量或框架魔法
- **接口与实现放在一起** — 当类型和实现在同一个文件中时，模型能看到完整的契约

### 为 AI 可读性命名

- 函数和变量名是提示词的一部分，即使你没有意识到
- 用 `parse_transaction_from_csv` 而不是 `parse_data`；更丰富的名字让模型做出更好的决策
- 代码库中一致的命名模式会叠加——模型学习你的规范并加以复现

### 上下文锚点

创建在开始新任务时给 AI 定向的文档：

- **`ARCHITECTURE.md`** — 模块地图、关键设计决策、使用的模式
- **`CLAUDE.md` 或 `AGENTS.md`** — 项目特定规范、需要避免的事项、如何运行测试、各种注意事项
- **接口文件** — 一个导出模块所有公共类型和签名的单一文件；在请求实现之前将其粘贴为上下文
- **模式示例** — 一个关于如何处理错误、分页、认证的规范示例——可以粘贴并说"按照这个模式"

## 代码提问的模式

### 明确提供约束

AI 会做出选择，除非你告诉它不能这样做：

> "使用现有的 `db.Query` 抽象来实现——不要引入新的 ORM 或查询构建器。"

> "匹配 `internal/api/handlers.go` 中的错误处理模式——用上下文包装错误，永远不要吞掉它们。"

> "不允许引入新依赖，只使用标准库。"

约束比描述更重要。描述模糊但约束严格的提问，比描述详细但没有约束的提问产生更好的代码。

### 引用现有模式

> "按照 `internal/org/service.go` 中 `DeleteOrganization` 的实现方式实现 `DeleteUser`。"

这比从头描述你想要什么更可靠。模型将现有代码视为模板并复现其中的模式——错误处理、日志记录、事务管理，一应俱全。

### 先定义签名再要实现

```go
// ProcessPayment 尝试向存档的银行卡收取给定金额。
// 如果因余额不足导致扣款失败，返回 ErrInsufficientFunds。
// 如果银行卡已过期，返回 ErrCardExpired。
// 所有其他错误应附带上下文包装后原样返回。
func ProcessPayment(ctx context.Context, userID string, amount Money) error
```

粘贴这段内容并让模型实现它。你已经定义了契约；它来填充函数体。这也迫使你在实现之前思考失败场景。

### 同时要求测试

> "实现这个函数，并编写表驱动测试，覆盖：正常路径、docstring 中的每个错误情况，以及你识别出的至少两个边界情况。"

与实现同时生成的测试比事后编写的测试更可能与实现保持一致。

### 要求解释

> "实现这个，然后解释：你做了什么假设？在规模化场景下什么可能出错？如果有更多时间你会怎么改？"

这能在隐藏的假设变成生产 Bug 之前将其暴露出来。

## 大规模上下文管理

上下文管理是规模化 Vibe Coding 的核心工程挑战。模型的上下文窗口是有限的；你的代码库不是。

### 策略

**按文件工作，而不是按功能工作。**
一个功能会涉及多个文件。试图在一个提示词中实现所有内容会产生不连贯的输出。分解为：先类型，再接口，再实现，再测试——每个步骤一个专注的提示词。

**显式总结上下文。**
不要假设模型知道它在什么环境中工作。每次会话开始时简短地介绍一下：

> "这是一个处理支付的 Go 服务。`internal/payment` 包拥有所有支付逻辑。我们通过 `db.Query` 使用 PostgreSQL，错误用 `fmt.Errorf` 包装，所有处理器遵循 `handlers.go` 中的模式。我接下来要让你实现 X。"

**维护一份持续更新的上下文文档。**
保存一份 `CLAUDE.md` 或 `AI_CONTEXT.md`，在复杂会话开始时粘贴。包括：项目做什么、关键模式、当前进行中的工作，以及需要避免的事项。随项目演进持续更新。

**将模型的输出作为下一个提示词的输入。**
生成接口后，将接口粘贴到生成实现的下一个提示词中。生成实现后，将其粘贴到测试生成提示词中。每一步都为下一步提供准确的上下文。

**将长任务拆分为检查点。**
对于跨多文件的功能：实现一个组件，审查它，提交它，然后继续。不要让 AI 一次生成十个文件——第一个文件中的错误会传播到所有建立在其上的内容中。

## 质量控制与代码审查

**永远不要合并你不理解的 AI 生成代码。**

这听起来显而易见，却屡屡被违反。输出很流畅，能编译，测试通过，审查者匆匆浏览一遍。六个月后出现生产事故，根源是某个细微错误，没有人发现，因为没有人真正读过那段代码。

### AI 生成代码中需要重点审查的内容

**错误处理** — 每个错误都被处理或显式传播了吗？AI 经常生成忽略错误或用注释吞掉错误的代码。

**边界情况** — 空输入、零值、nil/null、并发访问、超大输入。AI 测试正常路径；你测试边界。

**安全边界** — 任何涉及用户输入、文件路径、SQL 查询、认证令牌或权限级别的代码，都需要带着对抗性思维进行人工审查。AI 不会考虑攻击者。

**资源管理** — 连接、文件和协程是否被正确关闭？AI 经常在错误路径中遗漏清理操作。

**性能特征** — 循环中是否隐藏着 N+1 查询？这个算法在你实际会遇到的输入下会不会严重退化？

### 审查流程调整

将代码审查的重点从"语法是否正确"（模型处理这个）转移到"架构是否正确且安全"。

让模型审查自己的输出：

> "这个实现最薄弱的部分是什么？如果有更多时间你会怎么改？"

让它攻击自己的输出：

> "如果你试图让这段代码在生产环境中失败，你会利用什么？"

这些提示词能暴露出标准审查经常遗漏的问题。

## 迭代工作流

好的 Vibe Coding 不是单次提示词，而是结构化的迭代。

```
1. 规格说明   → 在触碰实现之前写好接口、docstring 或规格
2. 生成       → 带完整上下文和明确约束地提问
3. 阅读       → 真正地读输出；不要只是运行它
4. 质疑       → 让模型解释假设和薄弱点
5. 测试       → 运行现有测试；为新代码生成新测试
6. 精炼       → 用追问修复审查中发现的问题
7. 集成       → 验证新代码符合周边代码库的模式
8. 提交       → 用描述变更的提交信息，而不是 AI 会话的描述
```

最重要的步骤是第 3 步。跳过阅读输出的开发者，正在积累一笔用生产事故偿还的债务。

**把追问视为一等工作**，而不是纠错。"收窄""质疑""重新表述"对代码同样适用：

- *收窄* — "你处理了数据库错误，但没有处理网络超时。补充超时处理。"
- *质疑* — "你说这是 O(n)——带我走一遍为什么。"
- *重新表述* — "那个实现把缓存耦合到了数据库。重构为将缓存作为接口参数传入。"

## 团队实践

没有团队规范的 Vibe Coding 会产生不一致的代码库，不同部分看起来像是不同人写的——因为它们确实是，通过使用不同提示词的不同 AI 会话。

### 共享规范

像编写代码风格指南一样编写**团队提问指南**，应当规定：

- 始终提供什么上下文（架构摘要、关键模式）
- 始终包含什么约束（已批准的依赖、错误处理规范）
- 不让 AI 做什么（安全关键代码、未经审查的数据库迁移）
- 如何记录 AI 生成的部分（或根据团队政策决定是否记录）

### 代码审查文化

- 审查者应该能理解每一行——"AI 写的"不是解释
- 标记作者明显不理解的生成代码；退回重写
- 奖励理解，而不是速度

### 共享上下文文档

将 `ARCHITECTURE.md` 和 `CLAUDE.md`（或等效文档）作为仓库中的一等制品维护。保持更新。过时的上下文文档会误导人类和 AI。

### 结对 AI 会话

两个人加一个 AI 对复杂问题效果很好。一个人提问，另一个审查和质疑。AI 不替代结对——它加入结对。

## 常见陷阱

### 陷阱 1：不探索选项

**问题：** 接受第一个建议

**解决方案：**
```
始终问："还有其他方案吗？"
对比："给出我的场景的优缺点"
决策："因为【原因】，我们选【X】"
```

### 陷阱 2：改动太大

**问题：** 大型重构破坏了所有东西

**解决方案：**
```
应用 5 分钟规则
拆分为婴儿步骤
保持代码始终可运行
```

### 陷阱 3：没有备份

**问题：** 出问题时无法回滚

**解决方案：**
```
每次改动前都 git commit
可以对比新旧版本
轻松回滚
```

### 陷阱 4：上下文丢失

**问题：** 长会话中 AI 开始遗忘

**解决方案：**
```
定期刷新上下文
带摘要冷启动
保存会话笔记
```

### 陷阱 5：关注点混杂

**问题：** 组件职责过多

**解决方案：**
```
一个组件，一个职责
清晰接口
焦点窗口技术
```

### 陷阱 6：没有计划

**问题：** 直接跳到写代码

**解决方案：**
```
创建实现计划
与 AI 一起审查
按步骤实现
```

**Vibe 漂移** — 随着 AI 在不同会话中随意填充细节，代码库逐渐失去连贯性。修复：在每个提示词中明确强制模式；在模块层面审查一致性。

**规格债务** — 跳过规格说明快速推进，意味着 AI 设计了架构。修复：每次在生成实现之前都写规格说明。

**幻觉依赖** — AI 自信地使用不存在的库函数，或已废弃的 API。修复：始终对照实际文档检查导入和 API 调用。

**安全盲点** — AI 不会像攻击者一样思考。修复：所有安全敏感代码需要对抗性人工审查。

**测试剧场** — AI 生成通过但测试不充分的测试。修复：在生成测试之前写测试规格。

**所有权扩散** — 开发者对 AI 生成的代码感觉责任减少。修复：明确规定合并代码的人完全拥有它。

**上下文污染** — 长会话中早期错误传播到后续生成。修复：频繁提交有效的检查点；为不同组件开启新会话。

## 什么能规模化，什么不能

| AI 辅助效果好的 | 需要人类主导 |
|---|---|
| 在已定义接口内的实现 | 接口与架构设计 |
| 测试用例生成 | 测试策略与覆盖率决策 |
| 样板代码和脚手架 | 横切关注点规范 |
| 现有代码的文档 | 领域知识和业务规则 |
| 独立的重构任务 | 系统级重构 |
| 格式转换 | 安全审查 |
| 探索解决方案选项 | 最终设计决策 |

规律：AI 擅长处理**广度**——覆盖一个定义清晰的事物的所有情况。人类擅长处理**深度**——理解首先应该构建什么。

通过更善于定义问题来扩展 AI 使用。从 Vibe Coding 中获益最多的开发者，是那些在规格说明、架构和审查上投入更多的人——而不是提问更快的人。

## 总结：Vibe Coding 之道

### 核心原则

1. **探索方案** — 获取替代方案，比较，决策
2. **组件分离** — 单一职责，清晰接口，专注
3. **增量开发** — 微小改动，始终可运行
4. **会话管理** — 冷启动提供上下文，热启动保持流畅
5. **备份与计划** — 安全第一，先思考再写代码

### 日常工作流

```
早上：
- 带上下文启动会话
- 回顾今天的计划

白天：
- 为决策探索选项
- 做一个小改动
- 立即测试
- 如果可以就提交
- 重复

晚上：
- 创建会话摘要
- 保存备用
```

### 开发信条

```
决策前先探索
混合前先分离
求大前先求小
改动前先备份
编码前先计划
提交前先测试
```

### 记住

- AI 是你的伙伴，不是魔法
- 对话能揭示更好的解决方案
- 小步骤积累成大成果
- 能运行的代码胜过完美的代码
- 你随时可以继续改进

**掌握对话之道，掌握代码之道。**

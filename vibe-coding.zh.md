```
会话开始：  提供上下文 → 回顾计划 → 设定范围
每次变更：  小改动 → 测试 → 提交 → 重复
会话结束：  记录决策 → 保存上下文
```

## 什么是 Vibe Coding？

**Vibe Coding** 是一种协作式开发方式——你将 AI 助手作为技术伙伴，通过持续对话迭代地构建软件。

### AI 是你的技术伙伴

**不是魔法棒**——它是一个能够：
- 提出实现思路
- 解释权衡取舍
- 按你的决策实现代码
- 审查你的计划
- 帮助你思考问题

的对话伙伴。

### Vibe Coding 循环

```
1. 定义目标
   ↓
2. 探索方案
   ↓
3. 提供上下文
   ↓
4. AI 建议／实现
   ↓
5. 你审查并测试
   ↓
6. 对话：精化、质疑、迭代
   ↓
7. 下一个小步骤
```

### 找到"节奏"

在以下维度之间取得平衡：
- 足够的上下文 ↔ 不过度负担
- 具体明确 ↔ 创造空间
- 速度 ↔ 质量
- 接受建议 ↔ 坚持你的愿景

## 思维模式的转变

Vibe Coding 不是为了更快生成代码，而是转移你的认知投入方向。

传统开发：你同时拥有架构 *和* 实现细节。

规模化 Vibe Coding：你拥有架构、接口和约束；AI 在这些边界内处理实现细节。你的工作变成了**指导、审查和集成**，而不是编写代码。

这个转变听起来简单，实则不然。大多数开发者在 Vibe Coding 上失败，不是因为不会提示，而是因为没有真正内化哪些地方的人类判断是不可替代的：

- **你定义系统应该做什么** — AI 无法推断你的业务需求
- **你拥有架构** — 如果你不提供架构，AI 会发明一个；在代码规模到 10 万行时，你不会喜欢那个结果
- **你验证正确性** — AI 生成貌似合理的代码，不一定是正确的代码
- **你承载上下文** — 模型在会话之间会忘记一切；你不会

与 AI 协作效果最好的开发者，是那些变得更擅长写规格说明的人，而不是最信任 AI 输出的人。

## 适用场景与局限

### 高效用途

- **样板代码与脚手架** — CRUD 处理器、序列化器、CLI 参数、配置解析器
- **测试生成** — 良好定义函数的单元测试、表驱动测试用例、mock 脚手架
- **隔离的、有明确类型的函数** — 任何有清晰输入、输出且没有隐藏依赖的代码
- **有明确目标的重构** — "把这段逻辑提取成符合这个签名的函数"
- **格式转换与翻译** — 数据格式间转换、从描述生成 SQL、从类型生成 OpenAPI 规范
- **文档** — 生成文档注释、README 章节、从代码生成变更日志

### 容易出问题的地方

- **横切关注点** — 身份验证、日志记录、错误处理策略，这些需要在整个代码库中保持一致
- **复杂状态管理** — 任何正确性依赖于变更历史的代码
- **新颖算法** — AI 会产出能编译且看起来合理的东西，但可能存在细微错误
- **安全关键路径** — 输入验证、认证、加密、权限边界
- **需要理解你的领域的代码** — 训练数据中不存在的业务规则、数据不变量、法规约束
- **大规模重构** — AI 可以帮助执行步骤，但无法为线上系统设计安全的迁移路径

**经验法则：** 如果你很难为它写出规格说明，AI 也很难正确实现它。

## 核心原则

### 1. 组件分离
**一个组件，一个职责，干净的 API**
- 让 AI 保持专注
- 更易于测试
- 更易于理解

### 2. 增量开发
**微小变更，始终保持代码可运行**
- 5 分钟规则：如果验证一个变更需要超过 5 分钟，那就太大了
- 立即测试
- 频繁提交

### 3. 探索方案
**先获取多种实现思路，再做决策**
- 要求提供备选方案
- 比较优缺点
- 做出有据可查的决策

### 4. 会话管理
**冷启动时提供上下文，热启动时保持流畅**
- 新会话：提供完整上下文
- 活跃会话：保持连贯性
- 保存摘要供下次使用

### 5. 备份与计划
**安全第一，想清楚再写代码**
- 变更前始终备份
- 创建实现计划
- 与 AI 一起审查
- 逐步实施

## 对话思维

### 对话优于命令

❌ **较差：** "创建一个数据库类"

✅ **更好：**
```
你："我需要持久化数据，有哪些选项？"
AI：[解释 SQLite、PostgreSQL 等]
你："针对我的使用场景，各有什么优缺点？"
AI：[对比各选项]
你："我们用 SQLite 吧。API 应该是什么样的？"
```

### 关键思维转变

**从：** "告诉 AI 构建什么"
**到：** "与 AI 讨论如何构建"

**从：** "快速获取代码"
**到：** "先理解方法"

**从：** "大改动"
**到：** "微小、可验证的步骤"

**从：** "接受第一个答案"
**到：** "探索备选方案"

## 阶段感知型 Vibe Coding

不同的开发阶段需要与 AI 采用不同的交互策略。根据你在开发生命周期中所处的位置来调整你的方法。

### 规划阶段：探索性与战略性

**心态：** 最大化探索，理解权衡，着眼长远

**关键实践：**
- **提开放性问题**："[问题] 有哪些方案？"
- **要求多种方法**："给我 3 种解决方法"
- **评估权衡**："每种方法的优缺点是什么？"
- **考虑长期影响**："这个决策会如何影响未来开发？"
- **挑战假设**："这种方法可能出什么问题？"

**对话示例：**
```
你："我需要实现一个缓存层，有哪些选项？"
AI：[介绍 Redis、内存缓存、文件缓存等]
你："对于需要扩展到 100 万用户的系统，比较这些方案"
AI：[分析可扩展性、成本、复杂度]
你："每种方案的长期维护负担是什么？"
AI：[讨论运维复杂度]
你："我们用 Redis 来规划。如果从简单方案开始，迁移路径是什么？"
```

### 实现阶段：受控性与增量性

**心态：** 最小化变更，始终可运行，测试一切

**关键实践：**
- **只做最小变更**：将工作拆分为尽可能小的步骤
- **一次只做一个变更**：永远不要同时做多个修改
- **每次变更后测试**：立即运行测试，验证行为
- **使用版本控制**：保存前后状态以便对比
- **质疑变更**："为什么用这种方法？可能出什么问题？"

**示例工作流：**
```bash
# 1. 保存当前状态
git commit -m "变更前：添加缓存层"

# 2. 做一个最小改动
你："只添加缓存接口，还不实现"
AI：[只创建接口]

# 3. 立即测试
pytest tests/test_cache_interface.py

# 4. 对比版本
git diff HEAD~1 src/cache.py

# 5. 如果没问题则提交
git commit -m "步骤 1：定义缓存接口"

# 6. 下一个最小步骤
你："现在添加 Redis 实现，只做连接和断开"
```

### 验证与质疑：大型项目的关键

**对于大型软件包，要多次验证 AI 的回答。慢而彻底胜过快而错误。**

#### 验证原则

**AI 可能自信而错误。** 在大型项目中，一个错误会在整个代码库中级联放大。始终验证：

**质疑一切：**
```
你："实现用户认证"
AI：[提供实现]

你："解释这是如何处理密码哈希的"
AI：[解释]

你："bcrypt 是正确的选择吗？有什么替代方案？"
AI：[对比 bcrypt、argon2、scrypt]

你："你为什么选择 bcrypt 而不是 argon2？"
AI：[解释理由]

你："如果使用默认代价因子，安全风险是什么？"
AI：[解释代价因子的重要性]

你："向我展示如何用恶意输入测试这个"
AI：[提供安全测试]
```

**重复提问以确认：**
```
第一次：
你："这是如何处理并发请求的？"
AI："它使用互斥锁"

验证：
你："带我逐步走过两个请求同时到来时会发生什么"
AI：[逐步解释]

确认：
你："如果锁被持有太久会怎样？"
AI：[解释超时/死锁场景]

最终检查：
你："检查你的实现，有没有竞态条件？"
AI：[自我审查，可能发现问题]
```

#### 每次实现的验证检查清单

**在接受任何 AI 生成的代码之前：**

```markdown
## 准确性验证
- [ ] 请 AI 解释了方法
- [ ] 质疑了为什么选这种方法而不是其他方法
- [ ] 询问了边界情况
- [ ] 询问了失败模式
- [ ] 请 AI 审查了自己的代码
- [ ] 用不同方式提了相同问题（得到一致答案）

## 正确性验证
- [ ] 用正常输入测试
- [ ] 用边界情况测试（空值、null、最大值）
- [ ] 用恶意输入测试
- [ ] 验证了错误处理
- [ ] 检查了资源清理
- [ ] 审查了安全问题

## 集成验证
- [ ] 符合现有架构
- [ ] 遵循项目规范
- [ ] 与其他组件兼容
- [ ] 不破坏现有功能
- [ ] 性能可接受
```

#### 大型项目要慢下来

**不要着急，重复提问是好的实践：**

**❌ 快但有风险：**
```
你："添加缓存"
AI：[实现缓存]
你："看起来不错，提交"
[之后：缓存失效 bug、内存泄漏、竞态条件]
```

**✅ 慢但安全：**
```
你："添加缓存"
AI：[实现缓存]

你："解释缓存策略"
AI：[解释]

你："缓存满了怎么办？"
AI：[解释淘汰策略]

你："如何使缓存失效？"
AI：[解释失效机制]

你："如果两个线程同时失效会怎样？"
AI：[解释线程安全]

你："给我看这些场景的测试"
AI：[提供测试]

你："审查你的实现，有没有竞态条件？"
AI：[审查，发现潜在问题]

你："修复竞态条件"
AI：[修复]

你："解释这个修复"
AI：[解释]

[现在满怀信心地提交]
```

**花费时间：10 分钟 vs 2 分钟**
**防止的 bug：很多 vs 没有**
**节省的调试时间：数小时 vs 数分钟**

#### 验证的重复模式

**模式 1：解释-验证-确认**
```
1. 你："实现 [功能]"
   AI：[实现]

2. 你："解释这是如何工作的"
   AI：[解释]

3. 你："可能出什么问题？"
   AI：[列出风险]

4. 你："你是怎么处理 [具体风险] 的？"
   AI：[解释处理方式]

5. 你："在代码中指出那个处理在哪里"
   AI：[指向代码]

6. 你："有更好的方法吗？"
   AI：[可能会建议改进]
```

**模式 2：挑战-辩护-改进**
```
1. 你："你为什么用 [这种方法]？"
   AI：[给出理由]

2. 你："[替代方案] 怎么样？"
   AI：[对比]

3. 你："你的方法有 [潜在问题]，你怎么处理它？"
   AI：[解释或承认疏漏]

4. 你："修复这个问题"
   AI：[改进实现]

5. 你："解释这个修复"
   AI：[解释]
```

**模式 3：场景测试**
```
1. 你："[场景 1] 时会发生什么？"
   AI：[解释]

2. 你："[场景 2] 时会发生什么？"
   AI：[解释]

3. 你："[场景 1] 和 [场景 2] 同时发生时呢？"
   AI：[解释或识别问题]

4. 你："给我看这些场景的测试"
   AI：[提供测试]
```

#### 何时重复提问

**在以下情况重复提问：**

1. **回答看起来不完整**
   ```
   你："这是如何处理错误的？"
   AI："它捕获异常"
   你："捕获哪些具体异常？给我看代码"
   ```

2. **回答含糊**
   ```
   你："这是线程安全的吗？"
   AI："是的，它使用锁"
   你："哪些锁？在哪里？给我看"
   ```

3. **回答与之前的说法矛盾**
   ```
   之前："这是无状态的"
   之后："它缓存结果"
   你："等等，你说过无状态。解释这个缓存"
   ```

4. **你没有完全理解**
   ```
   你："解释这个算法"
   AI：[技术解释]
   你："用一个具体例子再解释一遍"
   AI：[基于例子的解释]
   你："用这个输入一步步走一遍：[具体输入]"
   ```

5. **风险很高（安全、数据完整性、金钱）**
   ```
   你："这是如何防止 SQL 注入的？"
   AI：[解释]
   你："给我看做到这一点的代码"
   AI：[展示代码]
   你："用这个恶意输入测试：[注入尝试]"
   AI：[测试]
   你："解释为什么它是安全的"
   AI：[解释]
   ```

#### 大型项目的关键原则

1. **慢下来才能快起来** — 10 分钟的验证节省数小时的调试
2. **重复提问** — 换种说法问同样的问题会暴露不一致性
3. **挑战假设** — "你确定吗？"是合理的问题
4. **要求证据** — "在代码中给我看"而不是"相信我"
5. **测试边界情况** — AI 针对正常路径优化，你负责测试边界
6. **质疑自信** — AI 即使在错误时也表现得很自信
7. **验证集成** — 它与系统的其余部分匹配吗？
8. **对抗性思考** — 攻击者能做什么？什么可能出错？

**记住：在大型项目中，一个未经验证的假设可能会级联成几十个 bug。花时间彻底验证。**

### 调试阶段：怀疑性与方法性

**心态：** 质疑一切，验证假设，避免大规模变更

**关键实践：**
- **质疑诊断**："你确定这是根本原因吗？"
- **验证假设**："给我看这是问题所在的证据"
- **要求最小修复**："测试这个理论所需的最小改动是什么？"
- **避免散弹枪式调试**：永远不接受多个同时发生的变更
- **比较前后**：保存两个版本以理解修复内容
- **测试修复**：验证它解决了问题且没有副作用

**对话示例：**
```
你："测试失败了。怎么了？"
AI："问题在数据库连接"
你："你怎么知道？给我看证据"
AI：[指向具体错误]
你："测试是否是那个原因的最小改动是什么？"
AI：[建议添加日志]
你："只添加日志，先不要修复任何东西"
[用日志测试]
你："确认了。现在最小的修复是什么？"
AI：[建议针对性改动]
你："只做那个改动，其他什么都不要改"
```

### 重构阶段：计划性与可回滚性

**心态：** 规划路径，保持功能，便于回滚

**关键实践：**
- **规划顺序**：开始前列出所有步骤
- **维护测试**：每一步都保持所有测试通过
- **保存检查点**：每次成功步骤后提交
- **验证等价性**：确保行为没有改变
- **便于回滚**：每个步骤应该可以独立回滚

**计划示例：**
```markdown
# 重构计划：提取缓存层

## 当前状态
- 缓存逻辑混在业务逻辑中
- 难以测试，难以替换实现

## 目标
- 干净的缓存接口
- 可插拔的实现
- 保持 100% 测试覆盖率

## 步骤（每步一次提交）
1. [ ] 提取缓存接口（不改变实现）
2. [ ] 为接口约定添加测试
3. [ ] 为现有代码创建适配器
4. [ ] 将第一个模块迁移到使用适配器
5. [ ] 验证测试仍然通过
6. [ ] 逐个迁移其余模块
7. [ ] 删除旧的缓存代码

## 回滚策略
- 每步独立可回滚
- 每步后必须测试通过
- 任何步骤失败，回滚并重新评估
```

### 使用存储的记忆与计划

**用文档控制范围：**

1. **会话上下文文档**：我们在做什么，什么不能动
2. **实现计划**：分步骤顺序，当前进度
3. **决策日志**：为什么选择这种方法，我们拒绝了什么
4. **已知问题**：什么坏了，现在忽略什么

**示例会话上下文：**
```markdown
# 会话：缓存层实现

## 范围内
- 缓存接口定义
- Redis 实现
- 缓存单元测试

## 范围外（不要动）
- 数据库层
- API 端点
- 认证逻辑

## 当前步骤
第 3 步，共 7 步：创建 Redis 适配器

## 下一步
1. 完成 Redis 适配器
2. 添加集成测试
3. 更新文档
```

### 版本对比工作流

**始终比较前后状态以理解变更：**

```bash
# 保存变更前状态
cp src/module.py src/module.py.before

# 用 AI 做变更
[AI 做修改]

# 对比
diff -u src/module.py.before src/module.py

# 或用 git
git diff src/module.py

# 自己审查变更
# 理解改了什么以及为什么
# 验证符合预期
```

### 跨所有阶段的关键原则

1. **你控制范围** — 用计划和上下文文档限制 AI 的变更
2. **始终最小化变更** — 永远不接受大规模修改
3. **立即测试** — 每次单一变更后验证
4. **保存版本** — 比较前后以理解影响
5. **质疑建议** — 尤其在调试时，验证诊断
6. **先规划再实现** — 想清楚步骤顺序
7. **便于回滚** — 每次变更应该可以独立撤销

## 先做架构，再 Vibe

规模化 Vibe Coding 中最大的错误，是从实现开始。

AI 非常擅长填充空间。如果你不给它架构，它会给你一个——从训练数据中最常见的开源项目派生而来。这可能不符合你的需求、你的现有模式或你的运营约束。

**在任何 AI 辅助实现之前，先做这些：**

1. **定义模块边界** — 每个组件的范围内外是什么；把它写下来
2. **先写接口定义** — 类型、函数签名、API 契约；这些成为 AI 必须满足的约束
3. **记录你的不变量** — 什么必须始终为真？你无法容忍的失败模式是什么？
4. **建立错误处理规范** — 错误如何传播，什么被记录，什么浮现给调用者；如果不指定，AI 会不一致地发明
5. **有意命名** — AI 将名称用作语义提示；模糊的名称产生模糊的实现

一个有用的练习：在生成任何代码之前，为你将要构建的模块写文档注释或规格说明。如果你不能清晰地写出来，你还没准备好实现它——无论有没有 AI。

## 为 AI 辅助构建项目结构

大型代码库有一个结构问题：任何给定任务的相关上下文分散在许多文件中，而 AI 的上下文窗口是有限的。

设计你的代码库，使其对只能一次看到几千个 token 的模型来说是可导航的。

### 文件和模块设计

- **保持文件小而专注** — 1500 行的文件对人来说难以推理，对模型来说更不可能完整保存在上下文中
- **每个文件一个概念** — 不要把不相关的抽象混在同一个模块中
- **显式导入优于隐式全局** — AI 可以看到导入的内容；它无法推断隐藏的全局变量或框架魔法
- **将接口与实现放在一起** — 当类型和实现在同一个文件中时，模型可以看到完整的契约

### 为 AI 可读性命名

- 函数和变量名是提示的一部分，即使你没意识到
- 优先用 `parse_transaction_from_csv` 而不是 `parse_data`；模型会用更丰富的名称做出更好的决策
- 整个代码库中一致的命名模式会产生复利效果——模型学习你的规范并重现它们

### 上下文锚文档

创建在开始新任务时给 AI 定向的文档：

- **`ARCHITECTURE.md`** — 模块地图、关键设计决策、使用的模式
- **`CLAUDE.md` 或 `AGENTS.md`** — 项目特定规范、需要避免的事项、如何运行测试、任何注意事项
- **接口文件** — 导出模块所有公共类型和签名的单一文件；在要求实现之前将其粘贴为上下文
- **模式示例** — 你处理错误、分页、认证的规范示例——可以粘贴并说"按这个模式来"的东西

## 组件分离与干净的 API

**黄金法则：** 保持组件分离，使用干净的 API，避免混淆 AI。

### 为什么重要

紧耦合会让你和 AI 都感到困惑：
- ❌ 职责不清晰
- ❌ 关注点混杂
- ❌ 难以测试
- ❌ AI 建议的准确性下降

✅ **解决方案：** 一个组件，一个职责，干净的接口

### 分离策略

**1. 一个组件，一个职责**

❌ **不好：**
```python
class DataProcessor:
    def process(self, data):
        # 验证、转换、保存、通知、记录……
        # 职责太多！
```

✅ **好：**
```python
class DataValidator:
    def validate(self, data) -> ValidationResult:
        """只做验证"""

class DataTransformer:
    def transform(self, data) -> TransformedData:
        """只做转换"""

class DataRepository:
    def save(self, data) -> SaveResult:
        """只做保存"""
```

**2. 干净的 API 边界**

❌ **不好：**
```python
def get_data(self):
    return self.db_connection  # 泄露了实现细节
```

✅ **好：**
```python
def get_data(self) -> List[DataRecord]:
    """返回干净的数据，隐藏获取方式"""
    return [DataRecord(...) for row in results]
```

**3. 最小依赖**

❌ **不好：**
```python
class ComponentA:
    def __init__(self):
        self.b = ComponentB()  # 紧耦合
        self.c = ComponentC()
```

✅ **好：**
```python
class ComponentA:
    def __init__(self, data_provider: DataProvider):
        self.provider = data_provider  # 接口，而非具体实现
```

### 焦点窗口技术

**一次只专注于一个组件：**

```
会话 1：设计并实现 DataSource
会话 2：测试 DataSource
会话 3：设计并实现 Validator
会话 4：测试 Validator
……以此类推
```

**在每个会话中：**
```
你："今天我们专注于 [组件 X]。
    这是它的接口：[粘贴]
    不用担心其他组件。"

AI：[保持专注于组件 X]
```

### 组件设计检查清单

✅ **清晰的输入/输出**
```python
def process(self, input: InputType) -> OutputType:
    """明确什么进什么出"""
```

✅ **无副作用**（尽可能）
```python
def transform(self, data: Data) -> TransformedData:
    """返回新数据，不修改输入"""
```

✅ **自包含**
```python
class Component:
    """内部包含所需的一切"""
    def __init__(self, dependencies: Dependencies):
        self.deps = dependencies
```

✅ **记录目的**
```python
class DataValidator:
    """
    验证数据记录。
    
    做：检查字段、类型、约束
    不做：转换、保存或通知
    """
```

### 危险信号

🚩 "这个类做太多事情了" → 拆分
🚩 "测试时需要设置一切" → 添加接口
🚩 "改 X 会破坏 Y 和 Z" → 解耦
🚩 "AI 总是混淆关注点" → 澄清边界

## 探索多种实现方案

**黄金法则：** 当有多种方式实现某功能时，先探索方案再做决策。

### 为什么重要

- ❌ 第一个建议可能不是最适合你的情况
- ❌ 你可能错过更简单的替代方案
- ❌ 权衡取舍并不总是显而易见
- ❌ 上下文决定正确选择

✅ **解决方案：** 系统性地探索方案

### 方案探索流程

**第 1 步：询问方案**

```
你："我需要 [解决问题]。有哪些不同的方法？"

AI："以下是主要方法：

    方案 1：[方法 A]
    - 描述：[是什么]
    - 最适合：[场景]
    
    方案 2：[方法 B]
    - 描述：[是什么]
    - 最适合：[场景]
    
    方案 3：[方法 C]
    - 描述：[是什么]
    - 最适合：[场景]
    
    你对哪个方向感兴趣？"
```

**第 2 步：请求优缺点对比**

```
你："针对我的具体使用场景比较这些方案：
    - [你的约束 1]
    - [你的约束 2]
    - [你的优先级]
    
    给我每种方案的优缺点。"

AI："针对你的使用场景：

    方案 1：[方法 A]
    优点：
    ✅ [优势 1]
    ✅ [优势 2]
    缺点：
    ❌ [劣势 1]
    ❌ [劣势 2]
    
    ……（其余方案类似）
    
    推荐：[方案 X]，因为 [原因]"
```

**第 3 步：深入了解不清楚的地方**

```
你："我在方案 A 和方案 B 之间犹豫，
    能详细说说 [具体方面] 吗？"

AI：[对该方面进行详细对比]

你："如果 [我的情况变化]，哪个更好？"

AI：[解释每种方案如何处理那个场景]

你："给我看每种方法的简单示例。"

AI：[提供代码示例对比]
```

**第 4 步：做出有依据的决策**

```
你："基于 [推理]，我们选方案 B。
    从最简单的实现开始。"

AI：[实现所选方案]
```

### 何时探索方案

**始终探索当：**
- 🔍 存在多个有效方法
- 🔍 性能很重要
- 🔍 可扩展性是顾虑
- 🔍 你对该领域不熟悉
- 🔍 决策有长期影响

**可以跳过当：**
- ✓ 实现很简单
- ✓ 适用标准模式
- ✓ 你已经知道最佳方法
- ✓ 之后容易修改

### 方案探索模板

```
你："我需要 [目标]。有哪些方案？"

AI：[列出 3-5 个方案及简短描述]

你："针对我的情况 [约束] 比较这些方案。
    给出优缺点。"

AI：[详细对比]

你：[如果仍不清楚] "详细说说 [具体方面]"
    或 "给我看方案 X 和 Y 的示例"

AI：[提供说明]

你："基于 [推理]，选方案 [选择]。"

AI：[实现]
```

## 代码提示模式

### 明确提供约束

如果你不指定，AI 会自行做选择。告诉它什么是不能做的：

> "用现有的 `db.Query` 抽象实现这个——不要引入新的 ORM 或查询构建器。"

> "按照 `internal/api/handlers.go` 中的错误处理模式——用上下文包装错误，永远不要吞掉它们。"

> "不添加新依赖。只用标准库。"

约束比描述更重要。有模糊描述但严格约束，比有详细描述但无约束，产出的代码质量更好。

### 引用现有模式

> "参照 `internal/org/service.go` 中 `DeleteOrganization` 的实现方式，实现 `DeleteUser`。"

这比从头描述你想要什么更可靠。模型将现有代码视为模板并重现这些模式——错误处理、日志记录、事务管理，一切都会跟上。

### 在实现前指定签名

```go
// ProcessPayment 尝试对存档的卡收取给定金额。
// 如果因余额不足而失败，返回 ErrInsufficientFunds。
// 如果卡已过期，返回 ErrCardExpired。
// 所有其他错误应带上下文包装后原样返回。
func ProcessPayment(ctx context.Context, userID string, amount Money) error
```

粘贴这个并要求模型实现它。你定义了契约；它填充函数体。这也迫使你在实现之前思考失败模式。

### 要求与实现同时生成测试

> "实现这个函数，并编写覆盖以下情况的表驱动测试：正常路径、文档字符串中的每个错误情况，以及至少两个你识别出的边界情况。"

与代码一起生成的测试，比实现后再写的测试更可能与实现保持一致。

### 要求解释

> "实现这个，然后解释：你做了哪些假设？大规模运行时可能出什么问题？如果有更多时间你会怎么做？"

这可以在潜在假设变成生产 bug 之前将其暴露出来。

## 增量开发

**黄金法则：** 做微小的增量变更，避免大规模重构。

### 为什么重要

大改动会带来问题：
- ❌ AI 丢失上下文
- ❌ 难以验证
- ❌ 难以调试
- ❌ 可能引入多个 bug
- ❌ 无法轻松回滚

✅ **解决方案：** 让改动小到显而易见的正确

### 5 分钟规则

**如果验证一个变更需要超过 5 分钟，那就太大了。**

❌ **太大：**
- "重构整个认证系统"
- "重写数据库层"

✅ **合适大小：**
- "为密码哈希添加一个辅助函数"
- "添加一个新的数据库查询方法"

### 增量策略

**1. 一次一件事**

❌ **不好：**
```
你："添加缓存、改进错误处理、优化算法"
AI：[做大量改动，把东西搞坏]
```

✅ **好：**
```
会话 1："为最后结果添加简单缓存"
[测试——通过！]

会话 2："让缓存大小可配置"
[测试——通过！]

会话 3："添加缓存过期"
[测试——通过！]
```

**2. 始终保持代码可运行**

❌ **不好：**
```
你："重写这个组件"
AI：[重写所有内容]
[什么都不工作，调试好几小时]
```

✅ **好：**
```
你："用新模式添加新方法，保留旧方法"
[测试——两者都工作！]

你："将一个调用方迁移到新方法"
[测试——仍然工作！]

你："迁移另一个调用方"
[测试——仍然工作！]

……继续直到完成，然后删除旧方法
```

**3. 婴儿步骤**

不要："重写验证系统"

而是拆分成：
1. 添加邮件验证规则
2. 测试邮件验证
3. 添加手机验证规则
4. 测试手机验证
5. 提取公共逻辑
6. 测试现有验证仍然工作

### 对话模式

```
你："这个变更看起来很大。能拆分成更小的步骤吗？"

AI："可以：
    1. [小步骤 1] - 5 分钟
    2. [小步骤 2] - 5 分钟
    3. [小步骤 3] - 5 分钟"

你："很好。我们只做步骤 1。"

AI：[实现步骤 1]

你：[测试——通过！]

你："好。现在步骤 2。"
```

### 增量检查清单

做出变更之前：

✅ 这是最小的可能变更吗？
✅ 能在 5 分钟内测试完吗？
✅ 现有代码还能运行吗？
✅ 能用一句话解释吗？
✅ 如果它出问题了，我能知道原因吗？

### 信条

```
做可能有效的最小变更
立即测试
如果有效，提交
然后做下一个小变更

永远不要：
- 重写大段代码
- 同时改多件事
- 破坏能运行的代码
```

## 大规模上下文管理

上下文管理是规模化 Vibe Coding 的核心工程挑战。模型的上下文窗口是有限的；你的代码库不是。

### 策略

**按文件工作，而不是按功能工作。**
一个功能会涉及很多文件。试图在一个提示中实现所有内容会产生不连贯的输出。将其拆分为：先写类型，然后是接口，然后是实现，然后是测试——每个都是一个专注的提示。

**明确地总结上下文。**
不要假设模型知道它在什么环境中工作。每次会话开始时做简短的定向：

> "这是一个处理支付的 Go 服务。`internal/payment` 包拥有所有支付逻辑。我们通过 `db.Query` 使用 PostgreSQL，错误用 `fmt.Errorf` 包装，所有处理器都遵循 `handlers.go` 中的模式。我即将让你实现 X。"

**维护一个活的上下文文档。**
保留一个在复杂会话开始时粘贴的 `CLAUDE.md` 或 `AI_CONTEXT.md`。包括：项目做什么、关键模式、当前正在进行的工作，以及需要避免的事项。随着项目演进更新它。

**将模型的输出用作下一个提示的输入。**
生成接口后，将该接口粘贴到生成实现的下一个提示中。生成实现后，将其粘贴到生成测试的提示中。每一步都为下一步提供准确的上下文。

**将长任务拆分为检查点。**
对于多文件功能：实现一个组件，审查它，提交它，然后继续。不要让 AI 一次生成十个文件——第一个文件中的错误会传播到其上构建的所有内容中。

## 文档与记忆管理

大型软件项目需要管理两种记忆：**短期记忆**（会话内）和**长期记忆**（跨会话和团队成员）。AI 可以帮助处理两者，但你必须设计记忆系统。

### 大型项目中的记忆问题

**AI 在会话间没有记忆。** 每次对话都是全新开始。对于大型项目：
- 你无法每次都重新解释整个架构
- 团队成员需要共享理解
- 几个月前做出的决策必须被记住
- 上下文的增长速度超过你能传达的速度

**解决方案：** 构建一个文档系统，作为人类和 AI 的外部记忆。

### 长期记忆：活文档

**创建随项目演进的持久上下文文档：**

#### 1. 架构记忆（`ARCHITECTURE.md`）

**目的：** 很少变化的高层系统设计

```markdown
# 系统架构

## 概述
```

## 会话管理

**黄金法则：** 新会话用冷启动，连续工作用热启动。

### 理解会话

**热启动（活跃会话）：**
- AI 有完整的近期上下文
- 快速，保持流畅
- 最适合：迭代工作、调试
- 限制：约 10-20 次交流

**冷启动（新会话）：**
- AI 没有记忆
- 需要重新建立上下文
- 最适合：新功能、新鲜视角
- 挑战：需要提供"记忆"

### 何时冷启动

需要新会话的迹象：
1. ✋ AI 忘记了之前的决策
2. ✋ 回答变得通泛
3. ✋ 当前会话已有 15+ 次交流
4. ✋ 转入不同阶段
5. ✋ 需要新鲜视角

## 质量控制与代码审查

**永远不要合并你不理解的 AI 生成代码。**

这听起来显而易见。它被常规性地违反。输出流畅；它能编译；测试通过；审查者粗略扫了一眼。六个月后，一个生产事故根植于一个细微的错误，没人发现，因为没人完整读过代码。

### 需要仔细审查的内容

**错误处理** — 每个错误是否都被处理或明确传播？AI 经常生成忽略错误或用注释吞掉错误的代码。

**边界情况** — 空输入、零值、nil/null、并发访问、非常大的输入。AI 测试正常路径；你测试边界。

**安全边界** — 任何涉及用户输入、文件路径、SQL 查询、认证令牌或权限级别的代码，需要用对抗性思维进行人工审查。AI 不会考虑攻击者。

**资源管理** — 连接、文件和协程是否正确关闭？AI 经常在错误路径中遗漏清理。

**性能特征** — 循环中是否隐藏着 N+1 查询？这个算法在生产中实际会遇到的输入上会严重降级吗？

### 调整审查流程

将代码审查的重点从"这在语法上正确吗"转移到"这在架构上正确且安全吗"。

要求模型审查自己的输出：

> "这个实现最薄弱的地方是什么？如果有更多时间你会改什么？"

要求它攻击自己的输出：

> "如果你想让这段代码在生产中失败，你会利用什么？"

## 迭代工作流

好的 Vibe Coding 不是单一提示，而是结构化迭代。

```
1. 规格说明  → 在接触实现之前写接口、文档注释或规格
2. 生成      → 带完整上下文和明确约束地提示
3. 阅读      → 实际阅读输出；不要只是运行它
4. 质疑      → 请模型解释假设和薄弱点
5. 测试      → 运行现有测试；为新代码生成新测试
6. 精化      → 用后续提示修复审查发现的问题
7. 集成      → 验证新代码符合周围代码库的模式
8. 提交      → 用描述变更的消息提交，而不是 AI 会话
```

最重要的步骤是第 3 步。跳过阅读输出的开发者正在积累一笔债务，他们将用生产事故来偿还。

**将后续提示视为一等工作**，而不是修正：

- *缩窄* — "你处理了数据库错误情况，但没有处理网络超时。添加超时处理。"
- *挑战* — "你说这是 O(n)——带我走过原因。"
- *重构* — "那个实现把缓存耦合到了数据库。重构以将缓存作为接口参数传入。"

## 安全优先的 Vibe Coding

**关键原则：** AI 生成能工作的代码，不是安全的代码。安全需要 AI 所缺乏的对抗性思维。

### AI 生成代码的安全问题

AI 在以下代码上训练：能在正常路径上有效、通过功能测试、对审查者来说看起来合理。

AI **没有**被训练去：像攻击者一样思考、考虑恶意输入、理解安全含义、识别细微漏洞。

**结果：** AI 生成的代码经常有功能测试无法发现的安全漏洞。

### 永远不要信任 AI 处理安全关键代码

**这些需要强制性的人工安全审查：**

1. **认证与授权** — 登录/登出逻辑、密码处理、会话管理、权限检查、令牌验证
2. **输入验证与净化** — 用户输入处理、文件上传、API 参数、数据库查询、命令执行
3. **数据保护** — 加密/解密、密钥管理、PII 处理、数据访问控制、审计日志
4. **外部接口** — API 端点、数据库查询、文件系统访问、网络调用、第三方集成

### AI 生成代码中的常见安全漏洞

#### 1. SQL 注入

**❌ AI 经常生成：**
```python
# 有漏洞 - 字符串拼接
query = f"SELECT * FROM users WHERE email = '{user_email}'"
cursor.execute(query)
```

**✅ 你必须强制使用：**
```python
# 安全 - 参数化查询
query = "SELECT * FROM users WHERE email = ?"
cursor.execute(query, (user_email,))
```

#### 2. 缺少输入验证

**❌ AI 经常生成：**
```python
def process_file(filename):
    with open(filename, 'r') as f:  # 路径遍历漏洞
        return f.read()
```

**✅ 你必须强制使用：**
```python
def process_file(filename):
    if '..' in filename or filename.startswith('/'):
        raise ValueError("无效文件名")
    safe_path = os.path.join(UPLOAD_DIR, os.path.basename(filename))
    with open(safe_path, 'r') as f:
        return f.read()
```

#### 3. 硬编码密钥

**❌ AI 经常生成：**
```python
API_KEY = "sk-1234567890abcdef"  # 永远不要这样做
```

**✅ 你必须强制使用：**
```python
import os
API_KEY = os.environ.get('API_KEY')
if not API_KEY:
    raise ValueError("未设置 API_KEY 环境变量")
```

#### 4. 缺少认证/授权

**❌ AI 经常生成：**
```python
@app.route('/api/user/<user_id>')
def get_user(user_id):
    return User.query.get(user_id)  # 没有认证检查！
```

**✅ 你必须强制使用：**
```python
@app.route('/api/user/<user_id>')
@require_authentication
def get_user(user_id):
    if not current_user.can_access_user(user_id):
        abort(403)
    return User.query.get(user_id)
```

#### 5. 错误信息泄露

**❌ AI 经常生成：**
```python
except Exception as e:
    return {"error": str(e)}  # 泄露内部细节
```

**✅ 你必须强制使用：**
```python
except Exception as e:
    logger.error(f"数据库错误：{e}")
    return {"error": "内部服务器错误"}
```

#### 6. 缺少速率限制

**❌ AI 经常生成没有速率限制的登录端点**

**✅ 你必须添加：**
```python
@app.route('/api/login', methods=['POST'])
@limiter.limit("5 per minute")
def login():
    ...
```

#### 7. 不安全的日志记录

**❌ AI 经常生成：**
```python
logger.info(f"用户登录：{username}，密码：{password}")  # 永远不要记录密码
```

**✅ 你必须强制使用：**
```python
logger.info(f"用户登录尝试：{username}")  # 无密码
```

### AI 生成代码的安全审查清单

```markdown
## 输入验证
[ ] 所有用户输入均已验证（类型、格式、范围）
[ ] 文件路径已净化（无路径遍历）
[ ] API 参数已根据 schema 验证

## SQL 与数据库
[ ] 所有查询使用参数化语句
[ ] 查询中无字符串拼接
[ ] 数据库错误不泄露 schema 信息

## 认证与授权
[ ] 受保护端点需要认证
[ ] 数据访问前检查授权
[ ] 密码处理使用正确的哈希（bcrypt、argon2）
[ ] 客户端代码中无认证逻辑

## 密钥与凭证
[ ] 无硬编码密钥、密码或 API 密钥
[ ] 密钥从环境变量或保险库加载
[ ] 密钥不记录，不在响应中返回

## 错误处理
[ ] 向用户返回通用错误消息（无细节）
[ ] 用户看不到堆栈跟踪

## API 安全
[ ] 公共端点有速率限制
[ ] CORS 配置正确
[ ] 强制执行输入大小限制
```

### 危险函数与安全替代品

#### Python

**❌ 绝不使用：**
```python
eval(user_input)        # 执行任意代码
exec(user_code)         # 执行任意代码
pickle.loads(data)      # 可以执行任意代码
os.system(f"ls {path}") # Shell 注入！
```

**✅ 使用替代品：**
```python
ast.literal_eval(user_input)          # 只求值字面量
json.loads(untrusted_data)            # 数据安全
subprocess.run(["ls", user_path])     # 无 shell 注入
```

#### JavaScript

**❌ 绝不使用：**
```javascript
eval(userInput)              // 执行任意代码
element.innerHTML = userInput // XSS 漏洞！
```

**✅ 使用替代品：**
```javascript
element.textContent = userInput              // 安全，无 HTML
element.innerHTML = DOMPurify.sanitize(html) // 已净化
```

#### 密码学

**❌ 绝不使用 MD5/SHA1 做密码哈希**

**✅ 使用：**
```python
import bcrypt
password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
# 或 argon2（推荐）
from argon2 import PasswordHasher
ph = PasswordHasher()
password_hash = ph.hash(password)
```

### 安全是人类的责任

**记住：**
- AI 可以实现你指定的安全功能
- AI 无法识别安全需求，无法进行对抗性思考，无法评估风险
- **你对 AI 生成代码的安全性负责**

**黄金法则：** 如果它涉及用户输入、认证、授权或敏感数据——假设它是有漏洞的，直到通过人工安全审查证明相反。

## 团队实践

没有团队规范的 Vibe Coding 会产生不一致的代码库，不同部分看起来像是由不同的人写的——因为它们确实是，通过有不同提示的不同 AI 会话。

### 共享规范

像写代码风格指南一样写**团队提示指南**。它应该指定：

- 始终提供什么上下文（架构摘要、关键模式）
- 始终包含什么约束（批准的依赖、错误处理规范）
- 不要让 AI 做什么（安全关键代码、未审查的数据库迁移）
- 如何记录 AI 生成的部分（或是否记录，基于团队政策）

### 代码审查文化

- 审查者应该能理解每一行——"AI 写的"不是解释
- 标记作者明显不理解的生成代码；退回重写
- 奖励理解而非速度

### 共享上下文文档

将 `ARCHITECTURE.md` 和 `CLAUDE.md`（或等价物）作为代码库中的一等工件来维护。保持它们是最新的。过时的上下文文档会误导人类和 AI。

### 结对 AI 会话

两个人加一个 AI 在复杂问题上效果很好。一个人提示；另一个审查和挑战。AI 不是要替代结对——它是加入结对。

## 常见陷阱

### 初级陷阱

**不探索方案** — 不问其他选择就接受第一个 AI 建议。修复：始终问"还有其他方法吗？"，在决策前比较优缺点。

**改动太大** — 尝试大规模重构导致一切崩溃。修复：应用 5 分钟规则，拆分成婴儿步骤，保持代码始终可运行。

**不备份** — 出问题时无法回滚。修复：每次变更前 `git commit`，比较新旧版本，便于回滚。

**丢失上下文** — 长会话中 AI 忘记了早期决策。修复：定期刷新上下文，带摘要冷启动，保存会话笔记。

**混杂关注点** — 组件承担太多职责。修复：一个组件一个职责，干净的 API，使用焦点窗口技术。

**没有计划** — 直接跳到代码不先思考。修复：创建实现计划，与 AI 一起审查，逐步实施。

### 高级陷阱

**Vibe 漂移** — 随着 AI 在不同会话中以不同方式填充细节，代码库逐渐失去连贯性。修复：在每个提示中明确强制执行模式；在模块级别进行一致性审查。

**规格说明债务** — 通过跳过规格说明快速推进意味着 AI 发明了设计。修复：每次在生成实现之前先写规格说明。

**幻觉的依赖** — AI 自信地使用不存在的库函数或已废弃的 API。修复：始终对照实际文档检查导入和 API 调用。

**安全盲点** — AI 生成在正常路径下有效的代码，在对抗性输入下表现很差。修复：所有安全敏感代码需要对抗性人工审查。

**测试剧场** — AI 生成能通过但测试覆盖不多的测试。修复：在生成测试之前写测试规格说明；以与实现相同的严格性审查测试。

**所有权扩散** — "AI 写的"成为隐性借口。修复：明确规定合并代码的人完全拥有它。

**上下文污染** — 早期错误传播到后续生成中的长 AI 会话。修复：频繁提交工作检查点；为不同组件开始新鲜会话。

## 哪些可以扩展，哪些不行

| AI 扩展良好 | 需要人类所有权 |
|---|---|
| 已定义接口内的实现 | 接口与架构设计 |
| 测试用例生成 | 测试策略与覆盖率决策 |
| 样板代码与脚手架 | 横切规范 |
| 现有代码的文档 | 领域知识与业务规则 |
| 隔离的重构任务 | 系统级重构 |
| 格式转换与翻译 | 安全审查 |
| 探索解决方案选项 | 最终设计决策 |

规律：AI 擅长处理**广度**——覆盖一个明确定义的事物的所有情况。人类擅长处理**深度**——理解首先要构建什么才是正确的。

通过更好地定义问题来扩展你的 AI 使用。从 Vibe Coding 中获益最多的开发者，是那些在规格说明、架构和审查上投入更多的人——而不是那些提示更快的人。

## 总结：Vibe Coding 之道

### 核心原则

1. **探索方案** — 获取备选方案，比较，决策
2. **组件分离** — 一个职责，干净的 API，专注
3. **增量开发** — 微小变更，始终可运行
4. **会话管理** — 冷启动提供上下文，热启动保持流畅
5. **备份与计划** — 安全第一，先思考再写代码

### 日常工作流

```
早上：
- 带上下文开始会话
- 回顾今天的计划

过程中：
- 为决策探索方案
- 做一个小改动
- 立即测试
- 有效则提交
- 重复

晚上：
- 创建会话摘要
- 保存供下次使用
```

### 信条

```
决策前先探索
混合前先分离
大之前先小
变更前先备份
编码前先计划
提交前先测试
```

### 记住

- AI 是你的伙伴，不是魔法
- 对话揭示更好的解决方案
- 小步骤积累成大成果
- 能运行的代码胜过完美的代码
- 你总是可以之后再精化

**掌握对话，掌握代码。**
